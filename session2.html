◆ Layer detail

Click a layer

Each layer only talks to its immediate neighbours. That isolation is what lets one app run on very different boards.

The same Python line works on a Pi and a BeagleBone. Only the bottom layers (driver + hardware) differ — and Yocto's BSP layer supplies those per board.
◆ Concept

Why the wall between kernel & userspace?

🧑‍💻 Userspace (your app)

Runs with limited privilege. It cannot touch hardware registers directly — a bug would crash the whole board. It must ask the kernel.

PythonTFLiteweb server

🛡️ Kernel (privileged)

Owns the hardware. It arbitrates between processes, enforces safety, and exposes a controlled door — the system call — for userspace to make requests.

driversschedulermemory
The doorway: a syscall like ioctl() switches the CPU into kernel mode, does the privileged work, then returns. That transition is cheap but not free — it's one reason precise timing is hard (Session 4).
🍼 In plain words You're a guest in a restaurant — you can't stroll into the kitchen and grab the knives. You ask the staff. The kitchen door (the syscall) is the only way in, and only the staff (the kernel) are allowed to hold the sharp tools (the hardware).
◆ Deep dive

What "privilege" actually means

The wall isn't a metaphor — it's enforced by the CPU itself.

Two hardware modes

ARM CPUs run code at different exception levels. Your app runs at a low level (EL0) where privileged instructions and raw device memory are simply illegal. The kernel runs at a higher level (EL1) where they're allowed. Try to touch a register from EL0 and the CPU faults — the kernel kills your process instead of letting it corrupt the system.

Virtual memory keeps you boxed in

Each process sees its own private address space via the MMU. Your pointers can't even name another process's memory or the hardware, so a bug in the car app can't scribble over Wi-Fi or the motor driver. Isolation = reliability.

🍼 In plain words The chip has a grown-ups-only switch. Your app is a toddler: reach for the sharp tools and the chip smacks its hand (a fault) and stops it — only the kernel gets grown-up powers. And every app is locked in its own sealed playroom (virtual memory), so one app tripping can't knock over another.

Anatomy of a system call

  1. Your app calls a C library function (e.g. ioctl(fd, …)).
  2. The library loads a syscall number + arguments into registers and executes the svc ("supervisor call") instruction.
  3. The CPU traps into the kernel at EL1 and jumps to the syscall handler.
  4. The kernel validates the request, does the privileged work (writes the GPIO register through the driver), and returns a result.
  5. Control returns to your app at EL0, as if the function simply returned.

That round-trip is measured in microseconds — negligible once, but it adds up if you do it thousands of times per second in a tight loop.

◆ The real files your app opens

In Linux, hardware looks like files

"Everything is a file" is Linux's superpower. Your car talks to hardware by opening special files under /dev and making syscalls on them.

🍼 In plain words Linux pretends every gadget is just a file. Want to talk to the motor chip? You open a file and write to it — exactly like scribbling in a notepad. The same open / read / write moves work for a pin, a sensor, or a plain text file.

Modern GPIO uses the character device /dev/gpiochip0. Your app opens it, requests specific lines, and sets their values — all via ioctl under the hood. The libgpiod library gives a clean Python API:

import gpiod
chip = gpiod.Chip("/dev/gpiochip0")
line = chip.request_lines({17: gpiod.LineSettings(
           direction=gpiod.line.Direction.OUTPUT)})
line.set_value(17, gpiod.line.Value.ACTIVE)  # pin 17 → HIGH → motor

The MPU6050 IMU sits on an I²C bus, exposed as /dev/i2c-1. You address the chip (0x68), then read/write its registers. smbus2 wraps the syscalls:

from smbus2 import SMBus
bus = SMBus(1)                 # /dev/i2c-1
bus.write_byte_data(0x68, 0x6B, 0)  # wake the IMU
hi = bus.read_byte_data(0x68, 0x43)   # gyro X high byte

Two wires (SDA + SCL) can host many devices, each at its own address — that's why I²C is everywhere in robotics.

Older code toggled pins by writing text to /sys/class/gpio/… (the sysfs interface). It's deprecated: slow, racy, and it can't atomically claim a pin.

The modern character-device interface (/dev/gpiochip* via libgpiod) is faster, safer, and lets one process own a line exclusively. We use it throughout the course.

sysfs = oldchardev = modern
This is the concrete answer to "how does my app reach the hardware?" — it opens a device file and makes syscalls. The library and kernel driver do the rest.
🍼 In plain words I²C is a two-wire party line: many gadgets share the same two wires, and each one has its own name (its address, like 0x68). You call out the name first, so everyone knows exactly who you're talking to.

◆ Build system

Declaring what the app needs

Our app imports gpiod and smbus2. If those aren't in the image, it crashes at start. The recipe declares them as runtime dependencies so BitBake guarantees they're installed.

car-training-app_1.0.bb
# runtime deps: present on the device, not just at build RDEPENDS:${PN} = "python3 python3-gpiod python3-smbus2" SYSTEMD_SERVICE:${PN} = "car-training.service" SYSTEMD_AUTO_ENABLE = "enable"

python3-gpiod → drives GPIO pins (motors)
python3-smbus2 → talks I²C to the MPU6050 IMU
python3 → the interpreter itself

meta-openembedded / meta-python
if a dep is missing…
$ ./main.py web ModuleNotFoundError: No module named 'gpiod' # fix: add it to RDEPENDS and rebuild the image
Build vs runtime: a build dependency (DEPENDS) is needed to compile; a runtime dependency (RDEPENDS) must be on the device. Our Python deps are runtime — a classic beginner trap.
🍼 In plain words DEPENDS = ingredients you need to cook the meal. RDEPENDS = things that must be on the table when you eat. Forget the fork (a runtime dep) and dinner fails at the table — even though the cooking went perfectly.
◆ Deep dive

How layers map onto the stack

Each layer of the Linux system is owned by a Yocto layer. That's the elegant symmetry of this whole course.

Linux layerProvided by (Yocto layer)Examples
Your applicationmeta-car-trainingthe car app + service
Librariesmeta-openembedded / meta-pythonlibgpiod, smbus2, numpy
C library + initpoky (core)glibc, systemd, BusyBox
Kernel + driverBSP: meta-raspberrypi / meta-yocto-bspSoC GPIO/I²C drivers
Bootloader + DTBSPPi firmware / U-Boot, .dtb
🍼 In plain words Every floor of the Linux building is built by one Yocto crew. The BSP (“board support package”) is the crew that knows this exact board — its pins, its chips, its wiring — so swapping to a different board mostly means swapping that one crew.

Layer priority

When two layers define the same recipe, the one with higher BBFILE_PRIORITY wins. This lets a board layer override a generic default without editing the original — powerful and dangerous.

.bbappend — extend, don't fork

To tweak someone else's recipe, drop a foo_%.bbappend in your layer. It appends to their recipe, so you customise the kernel or a package while staying upgrade-friendly.

🍼 In plain words You love a friend's recipe but want extra cheese. Instead of rewriting their whole card, you staple on a sticky note saying “+ cheese.” That sticky note is a .bbappend — you tweak their recipe without ever touching the original.

PROVIDES / RPROVIDES — the flip side of dependencies

A dependency is only satisfiable if some recipe provides it. The infamous Nothing RPROVIDES 'python3-libgpiod' error means: "you asked for a runtime package that no layer offers." The fix is to name the package that does exist (python3-gpiod) — or add the layer that provides it. Dependencies and provides are two ends of the same rope.

🍼 In plain words You ask the shop for “fizzy water,” but their shelf label says “sparkling water.” Same drink, wrong name — so the till says “we don't stock that.” The fix is to ask for the name that actually exists.
◆ Hands-on lab · ~40 min

See the layers for real (on the board)

  1. Hardware as files. List the device nodes the app uses: 
    ls -l /dev/gpiochip* /dev/i2c-*
  2. Scan the I²C bus and find the IMU at address 0x68: 
    i2cdetect -y 1        # 0x68 should appear in the grid
  3. Toggle a pin from the shell with the libgpiod CLI (no code needed): 
    gpioinfo | head            # list lines & who owns them
gpioset gpiochip0 17=1     # drive line 17 HIGH
  4. Watch a syscall happen. Trace the exact kernel calls the app makes: 
    strace -e trace=ioctl,openat python3 -c \
  "import gpiod; gpiod.Chip('/dev/gpiochip0')"
  5. Break it on purpose. Rename the gpiod module, start the app, and read the ModuleNotFoundError — then map it back to the missing RDEPENDS.
Aha moment: when you see openat("/dev/gpiochip0") and ioctl(...) scroll past in strace, you're literally watching your Python line become a kernel request. The whole stack, made visible.

◆ Common traps
Confusing DEPENDS with RDEPENDS. The app compiles fine but crashes on the board with a missing module — because you declared a build dep instead of a runtime dep.
Permission denied on /dev/i2c-1. The bus exists but your user lacks access. On the device you're usually root; off-device, add the user to the i2c/gpio groups.
Editing app code, image unchanged. Source is cached; run bitbake -c cleansstate car-training-app before rebuilding.
Using deprecated sysfs GPIO. Tutorials still show echo 17 > /sys/class/gpio/export. It's racy and slow — prefer libgpiod.
