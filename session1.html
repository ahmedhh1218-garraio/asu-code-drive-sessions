◆ Concept · the big architectural choice

Three ways to run code on a chip

◆ Words first

Five words we'll lean on — learn them before the concept:

Bare-metal
Running your code with no OS at all — your one program owns the whole chip. You do everything yourself.
Operating system (OS)
Software that manages the hardware and runs your programs for you. Linux is one. Think: the manager everyone reports to.
Kernel
The core of the OS that talks straight to the hardware and shares it between programs. The manager's brain.
Real-time
Guaranteeing an action happens within a strict deadline, every single time. RTOS & bare-metal can promise this; Linux cannot.
RTOS (real-time OS)
A tiny OS (FreeRTOS, Zephyr, QNX) that schedules tasks with guaranteed timing. A strict manager who never misses a deadline.

The stack on each option — watch it grow

Same chip; three very different amounts of software running on it. Each brick you stack up gives you a feature — and takes a little control and determinism away. Read every column bottom-up, from the metal to your code.

🔩 Bare-metal

Your application — high-level logic
while(1){ read_sensors(); if(clear) motor_forward(); }
🎚️ HAL — device actions
motor_forward(), motor_stop() — say what to do, not which pin
🔧 MCAL — register access
motor_forward() → set_pin(IN1, HIGH): writes the GPIO registers
🔌 The chip · metal
CPU + registers + memory-mapped IO
You get: µs-tight timing, tiny binary, total control.
You pay: no Wi-Fi, no multitasking, no filesystem, no Python — you write everything.

⏱️ RTOS

Your tasks
motor_task() · sensor_task() · comms_task()
+ RTOS kernel — the one extra layer
pre-emptive scheduler, timers, IPC (FreeRTOS / Zephyr / QNX)
🎚️ HAL — device actions
motor_forward(), motor_stop() — same semantic API
🔧 MCAL — register access
turns those actions into GPIO / timer register writes
🔌 The chip · metal
CPU + registers + memory-mapped IO
You gain: pre-emptive scheduling with guaranteed deadlines — perfect for a braking or motor-control loop.
Still missing: rich userspace — Python, TFLite, Wi-Fi, a dashboard are a huge lift.

🐧 Embedded Linux

Your app · Python + TFLite
main.py — ~100 lines of readable code
+ User-space libraries
python3, tflite-runtime, libgpiod, smbus2
+ Init & libc
systemd (PID 1), glibc, coreutils, ssh, journalctl
+ Linux kernel
scheduler, drivers, GPIO/I²C/USB, TCP/IP, filesystem
+ Bootloader + device tree
U-Boot / Pi firmware · .dtb describes the board
🎚️ HAL — kernel drivers & pinctrl
device tree + drivers map registers to /dev names — Linux's HAL
🔌 The chip · metal
CPU + registers + memory-mapped IO
You gain: multitasking, drivers, Wi-Fi, TCP, Python, TFLite, a filesystem — everything a rich AI app needs.
You pay: a bigger image and no hard real-time guarantee — the scheduler can pause your code for a moment.
🍼 In plain words Three ways to get around town: bare-metal is a go-kart — you push every pedal yourself, nothing between you and the wheels. RTOS is that same go-kart with a strict instructor who guarantees you brake in time. Linux is a comfy car with Wi-Fi, a screen and cruise control — powerful and clever, but it might pause for a heartbeat to take a phone call.

So why does our car run Linux?

Because its hard job isn't a microsecond control loop — it's running an AI model, streaming a dashboard over Wi-Fi, and logging data. That's exactly what a rich OS is good at. We accept Linux's one weakness — it's not real-time — and engineer around it. A pure control product might instead pick an RTOS; a self-learning, networked car picks Linux.

🏭 From the field

This is literally the infotainment vs. safety split in a real car

In production vehicles the infotainment & connectivity domain — maps, media, the AI, the network stack — runs embedded Linux, for exactly the reasons above. The safety-critical domain — braking, steering, airbags — runs on an RTOS or bare-metal microcontroller where a missed deadline is unacceptable. Same vehicle, two worlds, chosen by the real-time requirement. Our car is an "infotainment-class" workload, so Linux is the professional choice.

◆ Anatomy

What's actually on the SD card?

First — how did your code boot on the TivaC?

On a bare-metal microcontroller like the TivaC, power-on was simple: the chip's ROM boot sequence read the reset vector and jumped straight into main(), flashed at a fixed address. One program owned the chip — no OS, no filesystem, no "boot" to speak of. A Linux board cannot start like that: there's an entire operating system to load and configure first. So the SD card doesn't hold "your program" — it holds four stacked ingredients, and booting is each one handing off to the next.

Read the stack bottom-up — the order they hand control over at boot:

① Bootloader / firmware
Pi GPU firmware or U-Boot — the first code that runs, loads the kernel
first
② Linux kernel + device tree
The OS core, plus a .dtb that describes your board's hardware
core
③ Root filesystem (rootfs)
C library, BusyBox/coreutils, Python, systemd… the whole userspace
userspace
④ Your application
/usr/share/car-training + a systemd service that autostarts it
you

① The bootloader — what replaces TivaC's ROM jump

The SoC's tiny ROM can't start Linux directly — the kernel is compressed, large, and needs RAM configured first. So the ROM loads a small program whose only job is to prepare the machine and load the kernel: the bootloader.

On the Raspberry Pi that role is played by the GPU firmware (bootcode.bin → start.elf). On the BeagleBone it's U-Boot, the industry-standard open-source bootloader you'll meet on almost every embedded Linux board.

◆ What a bootloader does, in order
  1. Initialise RAM, clocks and the boot device
  2. Read its config — which kernel, which device tree, what boot arguments
  3. Copy the kernel image + .dtb into memory
  4. Jump to the kernel's entry point — then step aside for good
🍼 In plain words Turning the board on is like waking a sleepy giant. A tiny built-in reflex opens one eye and finds the SD card; the bootloader is the strong coffee that actually gets the giant (Linux) up on its feet.

② The kernel — and the device tree that describes your board

One Linux kernel binary runs on thousands of different boards. It can't hard-code "GPIO lives at address 0x7E200000" — that's true on a Pi and false on a BeagleBone. So the board's hardware is described separately, in a device tree.

◆ What is a device tree?

A device tree is a text file (.dts) that lists every piece of hardware on the board — CPUs, memory, I²C buses, GPIO controllers — and where each one lives in memory. It compiles to a binary blob (.dtb) that ships next to the kernel. At boot the kernel reads it and loads the matching drivers. Same kernel, different .dtb → different board.

◆ A device tree is literally a tree 
/ (root node)
├─ cpus/ .......... 4× Cortex-A72
├─ memory@0 ....... 4 GB
├─ soc/
│  ├─ i2c@7e804000  { status = "okay"; }   → /dev/i2c-1
│  ├─ gpio@7e200000 { ... }                → /dev/gpiochip0
│  └─ usb@7e980000  { ... }
└─ chosen { bootargs = "console=ttyS0"; }
Each enabled node becomes a device file the kernel exposes — the very /dev/* entries Session 2 will open. Want to switch a pin's function or turn on a sensor bus? You edit the device tree (or a small overlay), not the kernel source. It's the board's wiring diagram, in text.
🍼 In plain words The kernel is one skilled worker who can work in any factory. The device tree is the map of this factory — which machine sits where — so the worker finds everything without being rebuilt for each new building.

③ The root filesystem — everything Linux calls "userspace"

The kernel on its own does nothing useful — it needs programs to run. The rootfs is the whole directory tree mounted at /: the C library, shell utilities, Python, systemd, and finally your app. Why bake our own instead of grabbing Ubuntu? So it holds exactly what the car needs and nothing else — small, fast to boot, and reproducible byte-for-byte.

◆ Inside the rootfs 
/
├─ bin/  sbin/    # BusyBox / coreutils: ls, cp, mount…
├─ lib/           # glibc — the C library everything links to
├─ usr/bin/python3        # the Python that runs the AI
├─ usr/share/car-training/ # ← our app lives here
├─ etc/systemd/   # services, incl. car-training.service
└─ dev/ proc/ sys/ # kernel-provided, filled in at boot
◆ Ours vs a desktop

Desktop Ubuntu's rootfs is gigabytes — thousands of packages you'll never touch. Ours is a few dozen megabytes: every single file is present because a recipe put it there on purpose.

That's the whole promise of Yocto — you don't trim down a distro, you build up exactly the one you need.

🍼 In plain words The rootfs is the toy box holding every tool and program the system can use. A desktop ships a giant box crammed with toys you'll never touch; we pack a tiny box with only the toys our car actually plays with.

④ Your application — the only part that's truly "yours"

Everything below this layer was written by someone else. Your entire contribution is a handful of files dropped into the rootfs: the Python app under /usr/share/car-training/ and a systemd service that launches it automatically on boot.

That service is the bridge from "the OS finished booting" to "the car is driving": PID 1 (systemd) reads it and runs your main.py.

◆ Where your app sits at runtime 
systemd (PID 1)
  └─ starts car-training.service
       └─ /usr/bin/python3
            └─ /usr/share/car-training/main.py
                 └─ read sensors → model → drive motors
◆ Put the ingredients in motion

The full boot chain, stage by stage

You now know the four ingredients — so the boot sequence reads like a story. “It just boots” actually hides six distinct handoffs, each ingredient starting the next. Knowing them is the difference between guessing and debugging when the board shows a black screen.

  1. Power-on / ROM code. A tiny program burned into the SoC runs first — the same idea as the TivaC's reset vector, just smarter. It knows only enough to find the boot device (SD card) and load the next stage. You cannot change it.
  2. Bootloader. On the Pi it's the GPU firmware (bootcode.bin→start.elf); on BeagleBone it's U-Boot. Its job: initialise RAM, read config, load the kernel + device tree into memory, and jump to the kernel.
  3. Kernel + device tree. Linux decompresses and takes over. It reads the .dtb to learn what hardware exists, then brings up drivers, memory management and the scheduler.
  4. initramfs (optional). A tiny in-RAM filesystem that can run early setup — e.g. mounting an encrypted or network root. Our simple images often skip it.
  5. Mount root filesystem. The kernel mounts the real rootfs (the big partition) as /, then executes the first userspace process, PID 1.
  6. init = systemd. PID 1 is systemd. It brings the system up target by target and finally starts your car-training.service.
◆ Mental hook

Each stage only knows enough to start the next one. It's a relay race: ROM → bootloader → kernel → init → your app. If the car never reaches the dashboard, ask “which baton was dropped?”

Where problems show up

No lights at all → power/ROM. Rainbow screen forever (Pi) → firmware can't find the kernel. Kernel panic → bad rootfs or missing driver. Boots but no app → your service failed (check journalctl).

systemd targets — the "runlevels" of modern Linux

systemd boots by reaching targets, each pulling in the units it needs:

TargetMeaningAnalogy
basic.targetFilesystems, sockets, early services upEngine idling
network-online.targetWi-Fi has an IP addressRadio connected
multi-user.targetFull multi-user system, logins readyReady to drive

Our service declares After=network-online.target and WantedBy=multi-user.target so the dashboard only starts once Wi-Fi is up.

◆ The factory

What is Yocto — and why build a whole OS from source?

We never downloaded a finished operating system for the car. We build one, from source, tailored to our exact board. The tool that does that is Yocto — think of it as a factory that assembles a custom Linux distribution out of recipes.

◆ What Yocto actually is

Not a Linux distro you install — a build system. You describe what you want (which kernel, which libraries, your app) in text files, and Yocto compiles the whole thing from source into one flashable image. The same inputs always produce the same image, on any machine.

◆ Why Yocto for embedded
  • Tiny — only the packages the car needs, measured in MB not GB.
  • Reproducible — anyone rebuilds the identical image, bit for bit.
  • Tailored — the exact kernel drivers and device tree for our board.
  • Industry standard — the same tool shipping real products at real companies.
🍼 In plain words You don't buy a ready-made cake (that's Ubuntu). Yocto is your kitchen: you pick the exact ingredients and bake a cake that's precisely the size and flavour your car needs — no waste, nothing you didn't ask for.
◆ Words first

Yocto comes with its own vocabulary. Five words unlock everything that follows:

Image
The finished, flashable operating system — the single file you write to the SD card. The cake we're baking.
Recipe .bb
A text file with instructions to build one ingredient (the kernel, Python, our app). One recipe card per ingredient.
Layer meta-*
A folder (a Git repo) that groups related recipes. A shelf of recipe cards you stack in.
BitBake
The engine that reads the recipes and actually does the building. The oven.
Upstream / downstream
Upstream = the original project someone else maintains that we build on top of (poky, meta-raspberrypi). Downstream = us, further along, consuming their work. Water flows from upstream → to you, downstream.

What BitBake actually does — the five-task pipeline

BitBake builds every ingredient by running the same five tasks, in order. A full image is thousands of these, but each recipe walks the identical pipeline:

  1. do_fetch — get the source. Download this ingredient's code (git clone or tarball) into the build tree.
  2. do_configure — set it up. Run the project's configure step so it targets our board and options (think ./configure or cmake).
  3. do_compile — build it. Compile the source into binaries for the board's ARM CPU.
  4. do_install — lay out the files. Copy the built binaries, libraries and configs into the exact paths they'll occupy on the rootfs (a staging directory).
  5. do_package — wrap it up. Split the result into installable packages that get assembled, alongside every other recipe's, into the final image.
🍼 In plain words Building one ingredient is like making the same sandwich every time: fetch the bread, configure = lay it out, compile = toast it, install = plate it, package = wrap it to go. Every recipe walks the identical five moves.
Every task's output is cached (sstate). Change one recipe and BitBake re-runs only the tasks downstream of the change — which is why the second build is minutes, not hours.
◆ Why we use kas · not just “how”

Do it the hard way once — then let kas do it for you

BitBake alone works. It just needs you to babysit a lot of little setup steps every single time — clone each layer at the right branch, source an environment script, write two config files by hand, and add every layer to the build. Miss a step, get a cryptic error. kas is a thin YAML front-end that does the exact same steps for you. Look at both sides side by side and the point of kas becomes obvious.

🍼 In plain words Doing it by hand is reciting a six-step recipe from memory every single time — miss one step and dinner's ruined. kas writes the recipe on one card; hand the card to anyone and they make the exact same dish.

🛠️ Without kas — the manual dance

Everything you'd type in a terminal to reach one bitbake:

# 1) clone every layer at the same branch (typo = mystery error)
mkdir -p ~/yocto && cd ~/yocto
git clone -b scarthgap https://git.yoctoproject.org/poky
git clone -b scarthgap https://git.openembedded.org/meta-openembedded
git clone -b scarthgap https://git.yoctoproject.org/meta-raspberrypi
git clone <course-repo>/meta-car-training

# 2) initialise the build tree (sets PATH, creates build/)
source poky/oe-init-build-env build

# 3) register every layer with BitBake, in order
bitbake-layers add-layer ../meta-openembedded/meta-oe
bitbake-layers add-layer ../meta-openembedded/meta-python
bitbake-layers add-layer ../meta-raspberrypi
bitbake-layers add-layer ../meta-car-training

# 4) edit build/conf/local.conf by hand: MACHINE, DISTRO,
#    INIT_MANAGER, IMAGE_FSTYPES, EXTRA_IMAGE_FEATURES…
vim conf/local.conf

# 5) finally start the build
bitbake core-image-car-training
The pain: six manual steps, three files to keep in sync, and a teammate who wants to reproduce your build has to hear you explain all of the above. Version drift is guaranteed.

✨ With kas — one file, one command

The same steps, but declared once in YAML and executed for you:

# yocto/kas-raspberrypi4.yml — the entire recipe for the build
machine: raspberrypi4-64
distro: poky
target: core-image-car-training

repos:
  poky:
    url: https://git.yoctoproject.org/poky
    branch: scarthgap
  meta-openembedded:
    url: https://git.openembedded.org/meta-openembedded
    branch: scarthgap
    layers: {meta-oe: , meta-python: }
  meta-raspberrypi:
    url: https://git.yoctoproject.org/meta-raspberrypi
    branch: scarthgap
  meta-car-training:
    path: meta-car-training

  car: |
    INIT_MANAGER = "systemd"
    IMAGE_FSTYPES = "wic.bz2"

$ kas build yocto/kas-raspberrypi4.yml
The win: the YAML is the build. Anyone in the world runs one command and gets your exact image — no README, no tribal knowledge. That's the reproducibility professional Yocto projects live on.

What kas actually does under the hood

Nothing magic. When you run kas build it (1) git clones every repo in repos: at the exact branch/commit you asked for, (2) runs source oe-init-build-env, (3) generates bblayers.conf from the layer list, (4) appends your local_conf_header into local.conf, and (5) invokes bitbake $target. Exactly the manual dance on the left — automated and version-locked.

◆ Deep dive

Four more words that unlock the deeper mental model

You already met image, recipe, layer, BitBake and upstream/downstream in the primer earlier. Four more terms unlock everything else Yocto does behind the scenes — the ones you'll hear a senior engineer say out loud:

Task
One step of the five-task pipeline inside a recipe (do_fetch → do_configure → do_compile → do_install → do_package). BitBake runs them in dependency order — thousands of them for a full image.
Machine
The target hardware, e.g. raspberrypi4-64 or beaglebone-yocto. It selects the kernel, bootloader and device tree.
Distro
Global policy: init system, libc, features. We use poky tuned to systemd.
sstate cache
Yocto caches every task's output. Change one recipe and only that branch rebuilds — which is why the second build is far faster than the first.

Why build from source at all?

Because an embedded image must be tiny, reproducible and tailored. A desktop Ubuntu is gigabytes of stuff your car will never use. Yocto lets you include exactly the kernel drivers, libraries and files you need — nothing more — and rebuild the identical image on any machine, any time. That reproducibility is gold in a competition and in industry.

Where do these layers actually come from?

“Layer” sounds abstract. It isn't — a layer is just a Git repository full of text files (recipes and config). Here's where each one really lives on the internet:

LayerMaintained byOrigin (git URL)Gives us…

When you run kas build, kas reads those URLs from the .yml and git clones each layer next to the others. Afterwards your disk looks like this — every layer is a plain folder you can cd into and read:

tree -L 1 (after kas checks out the layers)
~/SummerTraining/build/layers/ ├─ poky/ # bitbake/ + meta/ + meta-poky/ ← from git.yoctoproject.org ├─ meta-raspberrypi/ # the Pi BSP layer ← from github.com/agherzan ├─ meta-openembedded/ # meta-oe, meta-python… ← from github.com/openembedded └─ meta-car-training/ # ← the ONLY one we wrote ├─ conf/layer.conf ├─ recipes-car/car-training-app_1.0.bb └─ recipes-car/files/app/main.py …

Why is each one required? Pull one out and the build breaks in a specific, predictable way:

Remove poky → no BitBake, no kernel, no C library. It's the ground everything stands on — nothing builds at all.
Remove meta-raspberrypi → no Pi kernel, firmware or device tree. The image might build but it will never boot on the board.
Remove meta-openembedded → no python3-gpiod / smbus2. BitBake stops with Nothing RPROVIDES because our app's runtime deps can't be found.
Remove meta-car-training → the OS boots perfectly… to an empty prompt. There's simply no car app on it. That's our layer.
Three of the four layers are downloaded unchanged from the internet. The only code that is truly ours is meta-car-training. Standing on giants' shoulders is the entire point of Yocto.
◆ Deep dive

How the layers link together

Cloning the folders isn't enough — BitBake has to be told which layers exist and in what order. Exactly two files do the linking.

① bblayers.conf — the master list

One variable, BBLAYERS, is an ordered list of every layer path BitBake should read. kas writes this for you from the .yml:

# build/conf/bblayers.conf  (generated by kas)
BBLAYERS = "
  ${TOPDIR}/layers/poky/meta
  ${TOPDIR}/layers/poky/meta-poky
  ${TOPDIR}/layers/meta-raspberrypi
  ${TOPDIR}/layers/meta-openembedded/meta-oe
  ${TOPDIR}/layers/meta-openembedded/meta-python
  ${TOPDIR}/layers/meta-car-training
"

② conf/layer.conf — each layer's ID card

Every layer carries its own conf/layer.conf that registers it with BitBake: its name, priority, where its recipes are, and which layers it needs first.

# meta-car-training/conf/layer.conf
BBPATH .= ":${LAYERDIR}"
BBFILES += "${LAYERDIR}/recipes-*/*/*.bb \
            ${LAYERDIR}/recipes-*/*/*.bbappend"
BBFILE_COLLECTIONS += "car-training"
BBFILE_PRIORITY_car-training = "10"
LAYERDEPENDS_car-training = "core openembedded-layer"
LAYERSERIES_COMPAT_car-training = "scarthgap"

Line by line — why each matters

BBFILES is the glob telling BitBake where this layer's recipes live. BBFILE_COLLECTIONS gives the layer a name. BBFILE_PRIORITY breaks ties when two layers define the same recipe (higher wins). LAYERDEPENDS says “I need these layers present first” — ours needs core (poky) and openembedded-layer for Python. LAYERSERIES_COMPAT pins the Yocto release (scarthgap) so mismatched layers refuse to build instead of failing cryptically later.

Confirm the wiring

On the build machine, ask BitBake what it actually loaded:

bitbake-layers show-layers
$ bitbake-layers show-layers layer path pri core layers/poky/meta 5 yocto layers/poky/meta-poky 5 raspberrypi layers/meta-raspberrypi 9 openembedded-layer layers/…/meta-oe 6 car-training layers/meta-car-training 10
Mental model of the whole chain: kas clones the layers → writes bblayers.conf to list them → each layer.conf plugs its recipes in and declares its dependencies. That's how six thousand recipes find one another.
◆ Read a real recipe

Anatomy of our app's recipe

This is (a trimmed version of) the recipe that puts the car software on the image. Every line earns its place.

# car-training-app_1.0.bb — packages the Python app + its service
SUMMARY = "Edge-AI line-following car controller"
LICENSE = "MIT"

# where the source comes from (a local folder in our layer)
SRC_URI = "file://app"

# runtime dependencies — MUST be present on the device, not just at build
RDEPENDS:${PN} = "python3 python3-gpiod python3-smbus2"

# install the app under /usr/share and register a systemd service
do_install() {
    install -d ${D}/usr/share/car-training
    cp -r ${WORKDIR}/app/* ${D}/usr/share/car-training/
    install -Dm0644 car-training.service \
        ${D}${systemd_system_unitdir}/car-training.service
}

inherit systemd
SYSTEMD_SERVICE:${PN} = "car-training.service"
SYSTEMD_AUTO_ENABLE = "enable"   # autostart on boot
${PN} = "package name", ${D} = the staging directory that becomes the rootfs, ${WORKDIR} = where the source was unpacked. These variables appear in every recipe.
◆ Reference

The files you write — and why

Yocto uses a small vocabulary of file types. Most come from upstream layers; you only author a handful. Here's the whole set and who writes each:

ExtensionWhat it isWho writes it
.bbRecipe — build/package one componentus (the app)
.bbappendExtends an existing recipe without copying itus, to tweak upstream
.confConfig: layer.conf, local.conf, machine/distrous (layer.conf) + upstream
.incInclude fragment shared between recipesmostly upstream
.bbclassReusable logic recipes inheritupstream (e.g. systemd.bbclass)

.servicesystemd unit — how/when our app startsus

You already read our recipe (.bb). Here's the systemd service it installs — the file that makes the car software launch on boot. Every line earns its place:

# car-training.service — a systemd unit (we write this)
[Unit]
Description=Edge-AI line-following car   # label shown by systemctl status
After=network-online.target            # don't start until Wi-Fi has an IP
Wants=network-online.target            # actively pull that target in

[Service]
ExecStart=/usr/bin/python3 /usr/share/car-training/main.py web
Restart=on-failure                     # crash? systemd relaunches it
User=root                              # needs GPIO/I²C access

[Install]
WantedBy=multi-user.target            # hook into the normal boot stage
[Unit]

Identity & order

Names the service and says when it may start. After/Wants put it behind the network so the dashboard has Wi-Fi before it opens a socket.

[Service]

How to run it

ExecStart is the exact command; Restart=on-failure gives resilience; User=root grants the privilege the pins need.

[Install]

When enabled

WantedBy=multi-user.target is what SYSTEMD_AUTO_ENABLE in the recipe hooks into, so the car autostarts every boot.

The thread that ties it all together

Our .bb recipe installs this .service file and our .py app into the rootfs; SYSTEMD_SERVICE + SYSTEMD_AUTO_ENABLE register it; systemd (PID 1) reads it at boot and runs our Python. Four small files — .yml, .bb, .service, .py — are the entire footprint of “our” work on top of a whole operating system.

◆ When it breaks (it will)

Troubleshooting field guide

Build fails: "Nothing RPROVIDES X".
A recipe needs a package no layer provides. Add the correct layer/package name (real example from this course: python3-libgpiod doesn't exist — the right one is python3-gpiod).
Board shows nothing / rainbow screen.
Firmware can't find or load the kernel. Re-flash, verify the card, and check the serial console for the first error line.
Boots but no dashboard.
The OS is fine, your service isn't. Run journalctl -u car-training — a missing Python module or wrong path is the usual culprit.
Edited the app but nothing changed.
The recipe caches source. Force a rebuild of just that recipe: bitbake -c cleansstate car-training-app && kas build …

Golden debugging rule

Always read the first error, not the last. Later errors are usually consequences. The serial console and journalctl are your two best friends all program long.

◆ No board? No problem

Boot the whole thing in QEMU — no hardware needed

Only one member of each team gets a physical board. Everyone else still needs to build, boot and hack on the exact same image — so they run it in QEMU: a machine emulator that boots your ARM image on an ordinary x86 laptop. Same bootloader, same kernel, same rootfs, same app — just no wires.

🍼 In plain words No real car to practise in? QEMU is a driving video-game that behaves just like the real car. You can crash it a hundred times and nothing actually breaks — perfect for the teammates without a board.
◆ What is QEMU?

QEMU emulates a whole computer in software — CPU, RAM, storage and peripherals. It reads the same kernel and rootfs the real board would, and executes the ARM instructions on your laptop by translating them on the fly. To the software running inside, it looks exactly like hardware.

◆ Why QEMU for this course?
  • No hardware barrier — the four teammates without a Pi build and boot the identical image.
  • Faster loop — edit → build → boot in a window, no flashing an SD card each time.
  • Crash-safe — a bad image just closes the window; nothing to re-flash or brick.
  • Same skills — the Yocto build, the recipes, systemd, journalctl — all identical.

The only thing that changes is the target machine — you point kas at a QEMU machine instead of the Pi. Everything else (target image, layers, your recipe) stays byte-for-byte the same:

🍓 For the real board

# kas-raspberrypi4.yml
machine: raspberrypi4-64
distro: poky
target: core-image-car-training
repos:
  meta-raspberrypi:      # ← BSP: Pi firmware + kernel
    url: …/meta-raspberrypi

Produces a .wic.bz2 you flash to a microSD.

💻 For QEMU

# kas-qemu.yml
machine: qemuarm64        # ← the only real change
distro: poky
target: core-image-car-training
# no meta-raspberrypi BSP needed —
# QEMU's machine ships with Poky

Produces an .ext4 + kernel that runqemu boots directly.

Then one command boots it — the same boot chain from earlier scrolls past, straight to a login prompt in a window:

runqemu — emulated serial console
$ kas build kas-qemu.yml # build the image $ runqemu qemuarm64 nographic # boot it, no window, serial only [ 0.000000] Booting Linux on physical CPU 0x0 [ 0.318402] Machine model: QEMU aarch64 Virtual Machine [ 1.204773] EXT4-fs (vda): mounted filesystem as / [ 2.244881] systemd[1]: Reached target multi-user.target [ 2.912004] systemd[1]: Started car-training.service. qemuarm64 login: root root@qemuarm64:~# _
The honest limit: QEMU has no real GPIO or I²C pins, so the motors and sensors are simulated (or the app runs in a headless test mode). It's perfect for the build, the boot and all the software logic — real hardware bring-up still needs the one physical board. Split the work: QEMU for everyone's dev loop, the board for the final wiring.
◆ Takeaways
  • Bare-metal, RTOS, Linux — pick by the real-time requirement. Our AI car needs rich userspace, so Linux wins (and we design around the jitter).
  • The car is a stack of layers. Every session opens one; keep the map in your head.
  • An image = bootloader + kernel + device tree + rootfs. Yocto/kas is the factory that builds it — reproducibly, from source, for our exact board.
